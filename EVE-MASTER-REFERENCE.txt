SECTION 2: ENGINEERING PRINCIPLES
======================================
These are the working rules built from real mistakes on this project.
Apply every session without exception.

BEFORE WRITING ANY CODE
------------------------
1. Run: git status — confirm clean working tree
2. Run: cat <file> to read the current file before editing
3. Identify EXACTLY which lines change — nothing else
4. Ask: will this break any other file that imports this?

MAKING CHANGES
--------------
5. One logical change per commit — never bundle unrelated edits
6. Always provide full file content — no ellipsis, no "// rest stays same"
7. After editing: npm run build locally before git push
8. If build fails: fix it before pushing — never push broken code
9. Never rename, move, or delete files without updating all imports

COMMITTING
----------
10. Always commit package-lock.json alongside package.json
11. Check git status after staging — confirm the right files are included
12. If git says "nothing to commit" after adding files, those files were
    already committed — check their actual content instead

DEBUGGING HIERARCHY (try in this order)
-----------------------------------------
1. Check git status — are changes actually committed?
2. Check Vercel build logs — did the deploy succeed?
3. Check browser JS chunks — is the code actually in the bundle?
4. Check console errors on live site
5. Check Supabase dashboard — are env vars right, are rows appearing?

SUPABASE-SPECIFIC
-----------------
- Two accounts exist — always verify URL matches the right project
- NEXT_PUBLIC_ prefix required for client-side env vars
- .env.local must NOT be committed (secrets stay local)
- Vercel env vars must be set separately from .env.local
- After adding env vars to Vercel, a new deployment must be triggered
- package-lock.json must be committed for Vercel to pick up new packages
- emailRedirectTo must be set on signUp or confirmation links go to localhost

REGRESSION PREVENTION
----------------------
- Read the file before editing it — never write from memory
- After any globals.css change: check fonts still load (Google import order)
- After any Nav.js change: check all 5 nav links still work
- After any auth change: test sign in AND sign up flows
- Never touch working mock data to "clean it up" — leave it until Supabase is wired

SECTION 3: DESIGN PRINCIPLES
======================================

EVE'S VISUAL IDENTITY
----------------------
- Tool aesthetic, not social app aesthetic — dense, purposeful, no fluff
- Forest green + warm white = serious creative workspace
- Playfair Display signals literary craft and gravitas
- No emojis, no illustrations, no stock photos — typography does the work
- Progress bars and stats are the "reward" — show work being done
- Tip boxes (green left border) = coaching voice, never nagging

PSYCHOLOGICAL DESIGN (dopamine + conversion)
--------------------------------------------
- Progress bars on every project card — completion anxiety drives engagement
- Stats on dashboard (3 projects, 17 scenes) — numbers feel real and earned
- "Resume ->" button language — implies momentum, not starting over
- Checklist beats in session mode — each tick is a small win
- "Act 2 of 3 complete" framing — always show where they are in the journey
- Lesson titles that provoke curiosity: "Nobody talks to have a conversation"
- Framework cards with "Best for:" copy — helps users self-select and commit
- "Start your first story" not "Sign up" — action language, not account language

CONVERSION PRINCIPLES (landing page)
--------------------------------------
- Hero shows the product, not a promise — the mock project card IS the sell
- Lead with craft credibility (Blake Snyder, Joseph Campbell, Dan Harmon)
- No-AI distinction is a feature, not a limitation — say it plainly
- Social proof goes here once real users exist (# of stories, # of scenes)
- CTA copy: "Start your first story" beats "Get started" every time
- Terms banner: unobtrusive, bottom of screen, one-time dismiss

WHAT MAKES EVE FEEL LIKE A TOOL (not a social site)

- Dense information layout — dashboard shows stats immediately
- No feed, no likes, no followers, no notifications
- Navigation is functional: Dashboard, Projects, Ideas, Session, Learn

PRO TIER ($9/month or $79/year — ~$6.59/mo):

  - All 4 session frameworks
  - Plot hole tracker
  - Themes & motifs tracker
  - Timeline / chronology view (once built)
  - Export to PDF / Final Draft (future)

HOW TO IMPLEMENT PAYMENTS
--------------------------
Use Stripe. It is the industry standard for SaaS.
1. Create Stripe account at stripe.com
2. Install: npm install stripe @stripe/stripe-js
3. Create two products in Stripe dashboard: Monthly + Annual
4. Add to Supabase: a `subscription_tier` field on profiles table
   ('free' | 'pro') and a `stripe_customer_id` field
5. Use Stripe Checkout (hosted page) — do NOT build your own payment form
6. After payment: Stripe webhook updates Supabase profile to 'pro'
7. Gate features in UI by checking user's subscription_tier

LEGAL REQUIREMENTS FOR CHARGING MONEY
--------------------------------------
These are non-negotiable once you take payments:
1. Terms of Service page (/terms) — required
2. Privacy Policy page (/privacy) — required (especially with Supabase/auth)
3. Cookie consent banner — required in EU, good practice everywhere
4. Stripe handles PCI compliance — do not build your own card handling
5. Refund policy — state it clearly in Terms (recommend: 7-day refund)
6. You will need to report this income on taxes (Schedule C or LLC)

PRICING PSYCHOLOGY
------------------

- Anchor with a crossed-out monthly price next to annual
- Add "Most popular" badge to annual plan
- Free tier is the hook — Pro is the unlock, not the gate

SECTION 5: METRICS TO TRACK
======================================

ACQUISITION (are people finding it?)
--------------------------------------
- Unique visitors per week (Google Analytics or Vercel Analytics)
- Sign-up conversion rate: visitors who create an account / total visitors
  Target: >3% is good for a cold landing page
- Source of traffic: direct, organic, paid, referral

ACTIVATION (do they get value fast?)
--------------------------------------
- % of new users who create a project within 24 hours
  Target: >40%
- % of new users who add at least 1 scene
  Target: >25%
- % of new users who complete a session beat checklist
  Target: >20%

RETENTION (do they come back?)
--------------------------------
- Day 7 retention: % of users who return 7 days after signup
  Target: >20% for a creative tool
- Weekly active users (WAU)
- Average projects per active user

REVENUE (once Stripe is live)
-------------------------------
- Free-to-Pro conversion rate. Target: >5%
- Monthly Recurring Revenue (MRR)
- Annual Recurring Revenue (ARR)
- Churn rate (% of Pro users who cancel per month). Target: <5%

HOW TO IMPLEMENT ANALYTICS (cheap/free)
-----------------------------------------
Option A (free): Vercel Analytics — built in, just enable in dashboard
Option B (free): Plausible or Fathom — privacy-first, writers will respect this
Option C (free): Add simple event tracking via Supabase
  When user creates project: insert row into analytics table
  When user completes session: insert row
  Query counts weekly in Supabase dashboard
